# Milestone M32.1: Frontend Feature Flags

## Summary
Implement a typed frontend feature flag system in `BuyAlan.WebApp` that works in SSR-rendered routes and client-side code, and leaves a clean extension path for future A/B testing without implementing A/B assignment in this milestone.

- Evaluate flags on the server for SSR-safe rendering.
- Hydrate the evaluated flag snapshot into client code.
- Make the same flag API available across landing, login, onboarding, and admin surfaces.
- Use a code-defined registry with environment-driven values for v1.
- Keep the evaluation and transport layers separate so future request-scoped or user-scoped experimentation can extend the system without changing consumers.

## Scope
- [ ] `BuyAlan.WebApp` only.
- [ ] App Router SSR usage in server layouts and server components.
- [ ] Client-side consumption through a shared provider and hooks.
- [ ] Typed flag registry and server-side evaluation.
- [ ] Hydration-safe snapshot transport from server to client.
- [ ] Tests for parsing, SSR behavior, hydration behavior, and client consumption.

## Non-Goals (Out of Scope)
- [ ] No A/B assignment, bucketing, stickiness, or experiment analytics.
- [ ] No backend API, WebApi, database, or schema changes.
- [ ] No remote config service integration.
- [ ] No per-user or per-request targeting in v1.
- [ ] No generated client or OpenAPI changes.
- [ ] No admin UI for managing flags.

## User Decisions (Locked)
- [x] V1 flag values come from a typed code registry plus environment/runtime configuration.
- [x] V1 evaluation is deployment-scoped only.
- [x] The flag system must be available across all app surfaces: landing, login, onboarding, and admin.
- [x] Future A/B testing must be enabled by architecture, but A/B itself is not part of this milestone.
- [x] No milestone scratchpad is required for this milestone.

## Findings from Repository Analysis
- [x] `BuyAlan.WebApp` uses Next.js App Router with separate root layouts for `(landing)`, `login`, `onboarding`, and `admin`.
- [x] There is no single global root layout already shared by all surfaces.
- [x] Server-side request data is already read in layouts via `cookies()` from `next/headers`.
- [x] Client-side shared state already uses provider patterns such as `SessionProvider` and `BreadcrumbProvider`.
- [x] There is currently no shared feature flag or runtime config layer in the frontend.
- [x] SSR-safe flag support must therefore be introduced as a shared pattern that each root layout opts into.

## Architecture Decisions (Locked)
- [x] Define flags in a typed registry under `src/lib/feature-flags`.
- [x] Resolve flag values on the server from environment variables, not directly in client code.
- [x] Hydrate a serializable flag snapshot into a client provider.
- [x] Expose server and client consumption APIs backed by the same snapshot shape.
- [x] Keep provider placement as deep as practical inside each root layout body instead of wrapping unrelated document structure.
- [x] Keep v1 flag values boolean-only.
- [x] Keep internal architecture open for future evaluator inputs such as cookies, user identity, or request context without committing to those inputs now.

## Public Interfaces / Contracts
- [ ] Add a typed `FeatureFlagKey` union derived from the registry.
- [ ] Add a serializable `FeatureFlagSnapshot` shape, keyed by `FeatureFlagKey`.
- [ ] Add server helpers:
  - [ ] `getFeatureFlagSnapshot`
  - [ ] `isFeatureEnabled`
- [ ] Add client helpers:
  - [ ] `FeatureFlagsProvider`
  - [ ] `useFeatureFlag`
  - [ ] `useFeatureFlags`
- [ ] Add a stable environment naming convention for flag values:
  - [ ] `FEATURE_FLAG_<UPPER_SNAKE_CASE_KEY>`

## Implementation Plan by Gate

## Gate A: Typed Registry and Evaluation Rules
- [ ] Create a central typed feature flag registry in `src/lib/feature-flags`.
- [ ] Define each flag with:
  - [ ] `key`
  - [ ] `defaultValue`
  - [ ] `description`
  - [ ] optional metadata fields that do not affect runtime behavior in v1
- [ ] Add parsing rules for env values:
  - [ ] `true`
  - [ ] `false`
  - [ ] `1`
  - [ ] `0`
- [ ] Fall back to registry defaults when env values are missing or invalid.
- [ ] Emit a server-side warning for invalid flag values without exposing sensitive runtime config.

### Gate A Acceptance Criteria
- [ ] Every flag is declared in one typed registry.
- [ ] Unknown flag keys are rejected by TypeScript.
- [ ] Missing or invalid env values produce deterministic defaults.
- [ ] Client code does not read raw env vars directly.

## Gate B: Server Snapshot Builder
- [ ] Add a server-side snapshot builder that resolves all registry flags into a serializable object.
- [ ] Add a server helper to evaluate a single flag from the same underlying snapshot logic.
- [ ] Keep the snapshot shape stable and framework-agnostic so future evaluators can reuse it.

### Gate B Acceptance Criteria
- [ ] Server components can evaluate flags through a shared API.
- [ ] A single snapshot source backs all server and client consumers.
- [ ] The snapshot is serializable and safe to pass through App Router boundaries.

## Gate C: Client Provider and Hooks
- [ ] Add `FeatureFlagsProvider` as a client component.
- [ ] Add `useFeatureFlag` for single-flag reads.
- [ ] Add `useFeatureFlags` for bulk reads when needed.
- [ ] Ensure hooks throw clearly when used outside the provider.

### Gate C Acceptance Criteria
- [ ] Client components can read flags through hooks.
- [ ] Provider state is initialized entirely from the server snapshot.
- [ ] No client-side refetch is required for v1.

## Gate D: Layout Integration Across All App Surfaces
- [ ] Integrate snapshot creation and provider wiring into the root layouts for:
  - [ ] `(landing)`
  - [ ] `login`
  - [ ] `onboarding`
  - [ ] `admin`
- [ ] Keep the integration pattern consistent across all route groups.
- [ ] Avoid introducing a fake global layout solely for flags.

### Gate D Acceptance Criteria
- [ ] The same feature flag API is available across all frontend surfaces.
- [ ] SSR-rendered UI can branch on flags before HTML is sent.
- [ ] Client components on the same routes see the same evaluated values after hydration.

## Gate E: Hydration Safety and Consumption Pattern
- [ ] Add a reference gated usage on at least one SSR-rendered page or shared component.
- [ ] Add a reference gated usage in at least one client component.
- [ ] Verify server-rendered and client-rendered output stay aligned when a flag changes markup.

### Gate E Acceptance Criteria
- [ ] A flag can hide or show SSR-rendered content safely.
- [ ] A client component can read the same flag without mismatch.
- [ ] No hydration drift is introduced by the feature flag layer.

## Gate F: Tests and Regression Coverage
- [ ] Add unit tests for env parsing and default fallback.
- [ ] Add unit tests for registry completeness and snapshot generation.
- [ ] Add client tests for provider and hooks.
- [ ] Add SSR-focused coverage for server snapshot usage.
- [ ] Add one browser-level regression scenario that verifies both disabled and enabled states for a test-safe flag.

### Gate F Acceptance Criteria
- [ ] Parsing and default behavior are covered by automated tests.
- [ ] Provider and hook behavior are regression-protected.
- [ ] SSR and hydrated client behavior are validated together.
- [ ] The initial rollout path for future flags is covered by at least one end-to-end scenario.

## Test Cases and Scenarios (Authoritative)
1. **Env parsing**
- [ ] `FEATURE_FLAG_SAMPLE=true` resolves to `true`.
- [ ] `FEATURE_FLAG_SAMPLE=false` resolves to `false`.
- [ ] `FEATURE_FLAG_SAMPLE=1` resolves to `true`.
- [ ] `FEATURE_FLAG_SAMPLE=0` resolves to `false`.
- [ ] Missing env uses the registry default.
- [ ] Invalid env uses the registry default and logs a warning.

2. **Server evaluation**
- [ ] A server helper can read a known flag by key.
- [ ] The generated snapshot contains every registered flag exactly once.

3. **Client provider**
- [ ] `FeatureFlagsProvider` exposes hydrated values to descendants.
- [ ] `useFeatureFlag` returns the expected boolean for a known key.
- [ ] Hook usage outside the provider fails clearly.

4. **SSR + hydration consistency**
- [ ] An SSR-rendered component gates content correctly when the flag is off.
- [ ] The same route hydrates without mismatch when a client component reads the same flag.
- [ ] Enabled and disabled states both render consistently.

5. **Cross-surface availability**
- [ ] Flag wiring works in at least one route under landing/login and one route under onboarding/admin.
- [ ] Root layout integration does not regress existing providers or route behavior.

## Risks and Mitigations
- [ ] Risk: duplicated wiring across multiple root layouts drifts over time.
- [ ] Mitigation: keep one shared snapshot/provider helper pattern and apply it uniformly.
- [ ] Risk: server-only env reads accidentally leak into client bundles.
- [ ] Mitigation: isolate env parsing and snapshot building in server-only helpers.
- [ ] Risk: hydration mismatch when SSR and client code evaluate differently.
- [ ] Mitigation: client code reads only the server-generated snapshot in v1.
- [ ] Risk: future A/B work becomes constrained by a boolean-only API.
- [ ] Mitigation: keep the evaluator and transport abstractions separate even though v1 returns booleans.
